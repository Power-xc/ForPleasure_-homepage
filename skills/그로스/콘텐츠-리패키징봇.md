# 콘텐츠 리패키징봇 스킬 (Content Repackaging Bot Skill)

> **담당:** 그로스부
> **Tier:** 1 (Claude Haiku)
> **Skill ID:** `content-repackage`

---

## 언제 쓰는가

- "이 글을 인스타/링크드인/X에 맞게 변환해줘"
- "초보자가 읽기 쉽게 다시 써줘"
- "이 글 약점 냉정하게 분석해줘"
- "뉴스레터 시리즈로 기획해줘"
- "임원 보고용 3줄 요약 만들어줘"
- `@content repackage {모드/원고}` 트리거 호출 시

> **콘텐츠-라이팅봇과의 차이:**
> - `@content write` → 새 글 생성
> - `@content repackage` → 기존 글 변환·편집·분석

---

## 실행 프로토콜

---

### Mode A — 멀티플랫폼 변환 (1원고 → 5채널)

입력: `원고 원문`

```
① 인스타그램
   - 어투: 감성적, 친근
   - 형식: 줄바꿈 많이 (2~3문장마다), 이모지 자연스럽게
   - 분량: 300자 내외
   - 해시태그: 10개 (도달 최적화: 대중 3 + 중간 4 + 니치 3)

② 링크드인
   - 어투: 전문적, 인사이트 중심
   - 형식: 첫 줄에 훅 → 본문 3~4문단 → 질문으로 마무리
   - 분량: 800~1200자 (알고리즘 권장 길이)
   - 금지: 해시태그 3개 초과, 외부 링크 첫 줄

③ 트위터/X
   - 분량: 140자 이내
   - 형식: 임팩트 있는 1문장 + 핵심 포인트 1가지
   - 스레드 확장 버전 (3트윗)도 함께 출력

④ 유튜브 설명란
   - 첫 3줄: 검색 키워드 포함 요약 (접기 전에 보임)
   - 타임스탬프 형식 (챕터별)
   - CTA: 구독/좋아요/댓글 유도

⑤ 카카오채널
   - 어투: 매우 친근, 구어체
   - 분량: 150자 이내
   - 형식: 이모지 1~2개 + 핵심 메시지 + CTA 버튼 문구

출력 후: "어떤 채널에 먼저 올릴지 우선순위 추천: {이유 포함}"
```

---

### Mode B — 독자 맞춤 리라이팅

입력: `원문 / 타깃 독자 정보`

```
설정 항목 확인:
  - 지식 수준: 초보자 | 중급자 | 전문가
  - 어투:      친근한 | 격식체 | 가볍게
  - 길이:      원문보다 짧게 | 비슷하게 | 길게
  - 전문 용어: 최대한 쉽게 풀기 | 그대로 유지

리라이팅 원칙:
  초보자: 전문 용어마다 괄호 풀이 + 비유 사용 + 단문 위주
  중급자: 기본 개념 설명 생략 + 응용 예시 추가
  전문가: 기술 언어 유지 + 근거/수치 밀도 높임

출력 후:
  원문과 리라이팅 버전의 차이점 3가지를 표로 정리
```

---

### Mode C — 글 비평 분석

입력: `평가할 글 원문`

```
냉정 평가 원칙: 칭찬은 짧게, 문제점은 구체적으로.

항목별 점수 (5점 만점) + 개선 버전:

① 구조와 흐름
   - 논리 순서가 자연스러운가?
   - 읽다가 '왜 갑자기 이게 나오지?' 하는 구간이 있는가?
   개선 버전: 섹션 재배치 제안

② 첫 문장
   - 3초 안에 읽을 이유를 주는가?
   - "안녕하세요" / "~에 대해 알아보겠습니다" 시작 → 즉시 -2점
   개선 버전: 대체 첫 문장 3가지

③ 근거의 구체성
   - "많은 사람들이" / "빠르게" 같은 모호한 표현 개수
   - 수치·사례·출처 밀도
   개선 버전: 모호한 표현 → 구체적 표현 치환 목록

④ 마무리 임팩트
   - 읽고 나서 뭔가를 할 것 같은가?
   - CTA가 명확한가, 흐지부지 끝나는가
   개선 버전: 마무리 단락 재작성

⑤ 가장 약한 부분 1개 (핵심 피드백)
   - 이것 하나만 고쳐도 퀄리티가 확 달라지는 것
   개선 버전: 해당 부분 전면 재작성

종합 평가: "{이 글의 가장 큰 문제는 ~이다. 고치면 ~이 달라진다.}"
```

---

### Mode D — 뉴스레터 시리즈 기획

입력: `전체 주제 / 기간(4주|8주|12주) / 독자 타깃`

```
출력 구조:

[시리즈 컨셉]
- 전체 제목 (독자가 구독하고 싶은 이유가 담긴)
- 독자가 시리즈를 다 읽으면 얻는 것 한 줄

[회차별 설계] (매 회차마다)
- 발행 순서: {n}회차
- 제목: 클릭하게 만드는 (15자 내외)
- 핵심 내용 요약: 3줄 이내
- 이전 회차 연결 고리: "지난 주에 ~를 배웠는데, 이번엔 그 다음 단계인 ~입니다"
- 다음 회차 예고: "다음 주엔 {제목}을 다룹니다 — {호기심 유발 한 줄}"

[시리즈 설계 원칙]
- 전체 흐름이 스토리처럼: 문제 인식 → 원인 이해 → 해결책 → 적용 → 심화
- 회차를 건너뛰어도 각각 독립적으로 읽힐 것
- 3회차마다 "지금까지 배운 것" 요약 회차 삽입 권장

오픈율 높이는 제목 공식 (회차별로 다르게):
  홀수 회차: 질문형 or 숫자형
  짝수 회차: 결과형 or 비밀 공개형
```

---

### Mode E — 3버전 요약

입력: `요약할 문서`

```
버전 1 — 트위터 요약 (140자 이내)
  규칙: 핵심 1가지만. 나머지는 전부 버린다.
  형식: [결론] + [왜 중요한가] in 140자

버전 2 — 임원 요약 (3문장)
  문장 1: 결론 (이 문서가 말하는 것)
  문장 2: 근거 (왜 이게 맞는가, 핵심 수치 1개)
  문장 3: 액션 (지금 당장 해야 할 것)
  규칙: 수동태 금지, 모호한 표현 금지

버전 3 — 상세 요약 (섹션별 핵심 포인트)
  형식: 불릿 포인트
  각 섹션마다: 핵심 주장 1줄 + 중요 수치/사례 1개
  마지막: "이 문서에서 가장 중요한 것: {한 줄}"
```

## Hard Gates

| ID | 조건 | FAIL 기준 |
|----|------|-----------|
| G1 | 원문 보존 | 핵심 주장 또는 사실관계가 변형됨 |
| G2 | 채널 적합성 | 채널별 길이, 톤, CTA 차이가 없음 |
| G3 | 버전 구분 | 요약/재작성/비평 결과물이 구분되지 않음 |

## Soft Findings

| ID | 조건 | 조치 |
|----|------|------|
| S1 | 플랫폼별 훅 | 각 채널 첫 문장 A/B 후보 제안 |
| S2 | 후속 제작 | 성과 좋은 버전을 copy 또는 email로 넘김 |
