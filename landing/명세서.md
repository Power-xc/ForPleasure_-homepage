## 0. 제품 이해 (앱 화면 분석 결과)

`images/`의 24개 앱 스크린샷을 분석해 도출한 제품 정체성.

- **브랜드:** 잇다 (Itda) — 슬로건 *"인연은 우연이 아니라, 잇다"*, *"사랑은 결국, 잇다"*
- **카테고리:** 검증 기반 **매니지드 데이팅**(결혼정보회사의 검증 + 앱의 편의성)
- **핵심 차별점 (앱 온보딩 6단계에서 추출):**
  1. **매칭이 끝이 아니라, 약속까지** — 연락처 교환 없이 앱 안에서 시간·장소까지 확정
  2. **선택은 신중하게, 매칭은 확실하게** — 여성이 먼저 남성 프로필을 보고 선택적으로 프로필 전송 → 남성은 이미 선택받은 상태에서 시작(거절·반복 시도 없음)
  3. **방치되지 않는 매칭 시스템** — 접속일수·응답속도·활동으로 비활성 회원 자동 제외, 끝까지 관리
  4. **직접 보고, 직접 검증** — 운영진이 오프라인에서 직접 유치·검증, 가입 심사로 선별된 풀
  5. **프리미엄은 그대로, 가격은 더 합리적으로**
  6. **노력하면 무료 만남도 가능** — 데일리 보상(루나) 지급, 활동 기반 무료 매칭
- **재화/시스템:** 루나(Luna, 활동 재화) · 만남권 · 잇다 매니저(컨시어지) · QR 프리패스(오프라인 특별 가입) · 친구초대 이벤트(루나 60개)
- **핵심 플로우:** 임시승인 → 스토리(자기소개·가치관, 최소 3개·심사) 입력 → 매칭 대기 → 프로필 열람 → 일정·장소 제안 → 채팅(약속 확정, 만남 전 연락처 교환 금지) → 약속 취소 시 패널티·환불

> 결론: 랜딩의 메시지는 *"아무나 만나는 앱이 아니라, 검증된 사람과 실제 만남까지 책임지는 앱"*. 신뢰·안전·검증이 1순위 소구점. [PROD][SEC]

## 1. 왜 만드는가 (목적)

현재 공식 홈페이지가 없다(Figma Make 링크는 "Create landing page" 플레이스홀더 상태). 잇다를 처음 접한 사람이 **무엇이 다른지 이해하고 → 가입/사전신청으로 전환**할 공식 입구가 필요하다.

---

## 2. 누가 쓰는가 (사용자)

```
주 사용자:
  - 진지한 만남을 원하지만 기존 데이팅앱의 피로(스팸·거절·연락 두절)에 지친 2030
  - 오프라인 행사/지인 추천으로 QR 프리패스를 받은 잠재 가입자
보조 사용자:
  - 제휴/입점 문의 파트너 (B2B)
  - 기존 회원의 친구초대 유입
```

---

## 3. 어떻게 쓰는가 (사용자 흐름)

```
1. 광고/검색/QR/지인공유로 랜딩 진입
2. 히어로에서 핵심 가치 1줄 + 대표 비주얼 인지
3. 스크롤하며 "왜 다른가" 6가지 차별점 + 실제 앱 화면 확인
4. 신뢰 근거(검증·안전·매니저) 확인으로 의구심 해소
5. 1차 CTA 클릭
   ├─ (확정 전) 사전신청/웨이팅리스트 폼 — 이메일·연락처
   └─ (출시 후) 앱스토어 다운로드 / QR 프리패스 가입 코드 입력
6. FAQ로 잔여 반론 해소 → 하단 CTA 재노출
```

> 분기: 모바일 진입(대다수) 시 스토어 버튼 우선, 데스크톱 진입 시 QR + 폼 우선.

## 4. 범위

### 이번에 하는 것 ✅ (v0.1)
```
- 반응형 원페이지 랜딩 (모바일 우선)
- 섹션: Hero / 문제제기 / 차별점(6) / 작동방식 / 신뢰·안전 / 이벤트 / FAQ / 최종 CTA / Footer
- 1차 전환 CTA = **QR 프리패스 가입** (초대코드 입력 → 가입 플로우 연결) [확정]
  └ 코드 없는 방문자용 보조 동선: "출시/가입 알림 받기"는 선택 — 9번 참고
- 앱 스크린샷(images/) 기반 비주얼
- 기본 SEO/OG 태그, 개인정보처리방침·문의 링크
```

### 이번에 하지 않는 것 ❌
```
- 로그인/회원 대시보드 등 앱 본기능 (앱 영역)
- 결제/실제 매칭 로직
- 다국어(i18n) — 한국어 단일
- 블로그/콘텐츠 허브 (추후 그로스 단계)
- A/B 테스트 인프라 (출시 후 funnel 스킬로)
```
